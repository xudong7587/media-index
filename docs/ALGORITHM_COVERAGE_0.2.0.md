# 0.2.0 后端算法覆盖说明

旧 `skill` 的作用是把判断步骤教给外部 Agent。0.2.0 已把这些步骤变成后端的确定性决策管线，因此运行 MediaIndex 不再依赖 Agent 阅读 skill，也不会把 TMDB Key 写在教学文件里。

## 能力映射

| 旧 skill / Agent 判断 | 0.2.0 后端实现 |
| --- | --- |
| 根据名称猜 TMDB 条目 | 后端以 TMDB ID 重新查询规范标题、原名、译名、年份和季 |
| 判断应追哪一集 | TMDB season episode、播出日期和逐集状态表 |
| 查看旧 QAS 链接 | 从任务状态或旧 QAS tasklist 恢复链接并读取真实文件目录 |
| 临时通过 PanSou 换源 | 目标集号、中文期数、播出日期、整季逐级搜索 |
| 判断候选是否靠谱 | 标题、季、年份、目标集证据、发布时间、衍生内容和链接有效性联合评分 |
| 从文件名识别集数 | SxxEyy、Eyy、中文集号、播出日期、唯一中文期数和上下期证据 |
| 避免综艺衍生内容 | 目标生成和文件匹配两层排除加更、纯享、花絮、预告、陪看等内容 |
| 生成 replace | 每个源文件生成锚定、转义、可预演的一对一重命名 |
| 执行 QAS | 只提交当前内存任务，执行后查询目标目录确认结果 |
| 不确定时询问用户 | 进入待确认、复用 QAS push_config 通知，用户可确认或重新搜索 |

## 为什么不直接让 LLM 每次判断

文件匹配和自动转存属于高风险、重复执行任务。确定性后端具有三个优势：

1. 同一输入会得到同一结果，可以写回归测试；
2. 每个分数和拒绝原因都能显示，便于追查误判；
3. 不会因为模型版本、上下文或提示词变化而突然采用更宽松的规则。

这里追求的“LLM 效果”是理解多种命名方式和上下文证据，而不是在生产环境中引入不可复现的自由推理。低置信度情况仍保留人工兜底，但不是常规流程。

## 仍需用真实样本验证的范围

- 同一天发布上下期、加更版和纯享版；
- 合并集，如 `E01-E02`、`EP01-02`；
- 文件名只有嘉宾或主题、没有集号和日期；
- 资源包包含多语言字幕、样片和重复清晰度；
- TMDB 集数维护错误或播出日期缺失。

正式升级前应把实际遇到的成功与失败文件名脱敏后加入回归样本。任何新规则都必须先通过旧样本，避免提高召回率的同时引入错误自动转存。
